# 用户管理系统设计

> 状态：待用户审核
> 日期：2026-07-12

## 目标

基于《乒乓球小程序技术方案》，补齐两类用户管理能力：

1. 小程序端普通用户中心，管理个人资料、运动数据、隐私和账号状态。
2. `manage/` 目录下的管理端，供管理员检索、查看和管理用户。

两者复用现有 NestJS API、JWT 登录和 SQLite/MySQL 数据模型，不改变已有场地、签到、约球、战绩业务的用户关联关系。

## 范围

### 小程序用户中心

- 查看头像、昵称、城市、打法和水平。
- 编辑昵称、头像、城市、打法和水平。
- 查看签到次数、打球时长、胜率、积分、成就等已有统计。
- 进入收藏、战绩、约球和签到记录。
- 修改隐私/通知偏好，退出登录。
- 对停用账号显示明确的不可用状态，不删除历史业务记录。

### 管理端

管理端源码统一放在根目录 `manage/`，不混入小程序 `src/` 页面。

- 管理员登录态校验和角色校验。
- 数据概览：用户总数、活跃用户、当前签到人数、待处理用户。
- 用户列表：按昵称/用户 ID 搜索，按状态、水平、城市筛选，分页。
- 用户详情：基本资料、账号状态、活跃数据、签到/约球/战绩摘要、管理员备注。
- 管理操作：启用、停用、调整角色、更新管理员备注。
- 停用和角色调整必须二次确认，并显示成功/失败反馈。
- 普通用户访问管理端页面或接口时返回无权限结果。

不在本期实现：复杂 RBAC 权限矩阵、批量导入导出、内容审核工作流、消息推送后台、审计日志检索页面。

## 数据模型

在现有 `users` 实体基础上补充：

- `city`: 用户所在城市，可为空。
- `style`: 打法风格，可为空。
- `level`: 水平等级，使用现有等级语义。
- `role`: `user` 或 `admin`，默认为 `user`。
- `status`: `active` 或 `disabled`，默认为 `active`。
- `lastActiveAt`: 最后活跃时间，可为空。
- `adminNote`: 管理员备注，可为空。

停用是软状态；已有签到、收藏、约球、战绩和成就记录不删除。停用用户不能继续调用需要登录的业务接口，管理员仍可查看其历史数据并恢复账号。

## API

普通用户：

- `GET /api/user/profile`: 返回当前用户资料和统计摘要。
- `PATCH /api/user/profile`: 校验并更新当前用户可编辑字段。
- `PATCH /api/user/preferences`: 更新隐私和通知偏好。

管理员：

- `GET /api/admin/users`: 分页查询用户，支持 `keyword/status/level/city/page/pageSize`。
- `GET /api/admin/users/:id`: 返回用户资料、统计摘要和最近业务活动。
- `PATCH /api/admin/users/:id/status`: 启用或停用用户。
- `PATCH /api/admin/users/:id/role`: 修改 `user/admin` 角色。
- `PATCH /api/admin/users/:id/note`: 修改管理员备注。
- `GET /api/admin/overview`: 返回概览统计。

所有管理接口同时经过 JWT 认证和管理员角色守卫；请求体使用 DTO 校验，用户不存在或状态不允许时返回明确的 4xx 错误。

## 数据流与错误处理

1. 普通用户登录后，JWT 的 `sub` 定位当前用户；资料接口只能读写自身数据。
2. 管理员登录后，JWT 角色声明和服务端数据库角色都必须为 `admin`，接口才继续执行。
3. 管理端遇到 401 清除本地 token 并回到登录态；403 显示“没有管理权限”；网络错误保留当前筛选条件并显示可重试提示。
4. 停用操作成功后刷新当前行和概览统计；操作失败不改变本地状态。
5. 管理端不直接读取数据库、不信任前端传入的用户角色。

## UI 方向

小程序端延续现有橙色品牌色、圆角白色面板和底部四栏导航，个人资料置于页面首屏，统计数据采用紧凑的横向数据带，设置项使用清晰的分组列表。

`manage/` 使用桌面优先的运营后台布局：左侧窄导航、顶部页面标题和管理员状态、内容区采用统计带 + 筛选条 + 用户表格。整体保持白底、深墨色文字、橙色主操作色和少量红色风险色，强调扫描效率，不使用营销式大幅卡片或装饰背景。

## 测试与验收

后端测试覆盖：

- 普通用户只能更新自身资料。
- 未登录和非管理员访问管理接口分别返回 401/403。
- 用户筛选、分页和详情返回稳定结构。
- 停用用户不能访问需要登录的业务接口，恢复后可以访问。
- 角色和状态更新不会删除历史业务记录。

前端测试覆盖：

- 用户中心展示资料和统计，编辑成功后更新页面。
- 管理列表搜索/筛选、打开详情、停用/恢复和角色修改流程。
- 401/403/网络错误的显示状态。

浏览器验收：

- `http://localhost:10086/` 首页和用户中心可访问。
- `manage/` 管理端可启动并显示概览页面。
- 普通用户无法进入管理端受保护页面。
- 管理员可完成用户搜索、查看详情和停用/恢复操作。

## 实施顺序

1. 先补齐后端用户字段、DTO、查询和管理员守卫。
2. 为后端接口写失败测试并实现通过。
3. 创建 `manage/` 管理端骨架和 API 客户端。
4. 完成管理端概览、列表和详情管理流程。
5. 完善小程序用户中心和资料编辑接口。
6. 启动前端与后端，执行接口测试和 companion 浏览器验收。
